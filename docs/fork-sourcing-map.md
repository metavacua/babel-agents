# Fork Sourcing Map — What's Taken From Where, and Why It's Translation, Not Verbatim Reuse

**Purpose:** name exactly which piece of which `metavacua` fork maps to which crate in this workspace, and be explicit about what's actually reusable as-is versus what requires translation/reinterpretation.

**Why translation dominates here, not verbatim copy:** every candidate fork (`adk-rust`, `juncture`, `langgraph-rust`, `covenant`, `swarms-rs`) was written assuming a full `std` environment — async runtimes, `tokio`, heap allocation via `std::sync::Arc`, OS threads. None target `wasm32v1-none`. Since every crate in this workspace must attempt to compile against that target, their actual source is rarely directly reusable line-for-line; what's reusable is the *design shape* — trait boundaries, state-machine structure, enum variants — reimplemented natively in `no_std` Rust. Where a piece's own license (Apache-2.0 for `adk-rust`/`juncture`/`swarms-rs`) would permit verbatim inclusion, its `std`-dependent implementation still has to be rewritten for this target; the license is not the blocker, the target is.

## What's genuinely NOT sourced from any fork

The dual-layer enforcement pattern itself (compile-time marker trait + runtime deny-by-default check, chassis design §6) — no candidate fork enforces both layers together; each does at most one. This is this project's own design, not adapted from any fork.

## Runtime choice, stated once

`wasmi`, not `wasmtime` — pure-Rust interpreter, no Cranelift/native-codegen dependency, portable to targets Cranelift doesn't support. `agent-orchestrator`'s `runtime` feature gates it (off by default, so the crate's own dispatch logic stays `wasm32v1-none`-clean; native CI jobs enable it explicitly).
